# 会话存档机制

**日期**：2026-04-11
**类型**：流程改进（非功能开发）
**跳过调研**：是——纯 harness 规则调整，无新依赖或交互模式

## 背景

Claude Code 单次会话上下文有限。当前 harness 的收尾/交接机制只覆盖"任务完成"场景，无法处理讨论/方案制定中途上下文耗尽的情况。

## 改动

1. **新建 `project-harness/context/working-draft.md`**
   - 中间态存档文件，用于保存未完成讨论的断点
   - 初始为空，存档后写入，任务完成后清空

2. **修改 `CLAUDE.md` 启动规则**
   - 新增 working-draft.md 作为启动时第一个读取项
   - 有存档时优先提示用户是否继续

3. **新增 `CLAUDE.md` 会话存档机制章节**
   - 中间态存档：定义存档内容格式（核心问题、方案、倾向、开放问题、下一步）
   - 会话切换自动提示：触发信号为"需要重读本会话早期已读文件"（上下文压缩的唯一可观测信号）

## 决策记录

- 最初设计了 4 个触发信号（轮次、文件数、子话题数、重读文件），经讨论精简为只保留"重读文件"，因为 Claude Code 不暴露 token 使用量、对话轮次等系统指标，其余信号均为不可靠的主观估算。
