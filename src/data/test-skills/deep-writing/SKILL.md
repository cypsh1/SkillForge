# Deep Writing

## 核心原则

1. **一次成稿**：默认单次产出完整文章，不做内部多轮迭代
2. **信息够就动手**：用户已给出主题、角度或素材时，直接写，不做诊断问卷
3. **先交付再打磨**：宁可先给用户一个完整但粗糙的稿子，也不在后台空转打磨

## 启动判断

收到写作请求后，快速评估（在思考中完成，不额外调工具）：

| 用户提供了什么 | 做什么 |
|---------------|--------|
| 主题 + 角度 + 素材/背景 | **直接写** |
| 主题 + 角度，素材不足 | 搜索补充（最多 2 次），然后写 |
| 只有模糊方向 | 问 1-2 个关键问题（不给菜单式选项表），等用户回复后再写 |

**禁止**：在用户意图已经清晰的情况下，自行生成选项再自行选择。

## 写作执行

### 第一步：确定结构

根据内容自然选择，不需要走决策树。常见模式：

- **观点先行**：核心论点 → 论据展开 → 收束（适合大多数分析/评论文章）
- **故事驱动**：场景/经历 → 冲突/转折 → 洞察/结论（适合叙事、个人体验类）
- **问题导向**：问题是什么 → 为什么 → 怎么办（适合指南、方案类）

不确定用哪种时，默认用「观点先行」。

### 第二步：一次性产出完整稿件

要求：
- 核心观点在前 10% 篇幅内出现
- 每个关键论点有具体证据（数据、案例、引用），而非笼统描述
- 段落首句是核心句
- 该用表格/列表提升可读性的地方就用

### 第三步：输出前自检

在输出前（同一次生成中）检查：
- [ ] 是否回应了用户的核心需求
- [ ] 有没有偏题或过度发散
- [ ] 有没有明显的事实性风险（不确定的信息标注来源或注明待核实）
- [ ] 有没有该删的冗余（套话、重复论证、过度铺垫）

自检发现问题就地修正，不另起一轮。

## 用户要求修改时

- 针对反馈做定向修改，不重写全文
- 如果用户要求「深化」「扩展」，只扩展指定部分

## 资源约束

- 单次写作任务的模型调用应控制在合理范围，避免不必要的内部循环
- 如需调研（搜索、读取 URL），控制在 3 次以内，够用就停
- 不要为了「理论品牌化」而强行造概念，除非用户明确要求

## 参考资料

写作类型详解、读者分析、自检清单等参考材料见 references/ 目录，需要时查阅，不需要每次都读取。
